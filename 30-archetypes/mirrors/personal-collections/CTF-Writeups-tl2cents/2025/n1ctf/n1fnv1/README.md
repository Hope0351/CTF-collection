## Core Idea

The challenge involves finding a preimage for a custom hash function: `x_next = ((x_prev * a + b + bit) ^ c) % m`, where `m = 2^128`.

The crucial vulnerability lies in the modulus `m` being a power of two. This means that the lower `k` bits of the hash state are only affected by the lower `k` bits of the previous state and the current input bit. Higher bits do not influence lower bits. This property allows for a progressive attack, where we can find a collision for the least significant bits (LSBs) first and then incrementally extend the collision to more significant bits without invalidating the already-collided lower bits.

The attack proceeds in stages, progressively matching the target hash from the LSBs up to the full 128 bits.

## Attack Steps

The strategy is as follows:

1.  **Initial LSB Collision (Stage 1):** We start by finding a short sequence of bits that makes the lowest 32 bits of the hash state match the lowest 32 bits of the target value. This is a classic search problem that can be efficiently solved using a Meet-in-the-Middle (MITM) attack (`find_lsb_path_mitm` function). We search for a path from the initial state `x_init` to the target state, considering only the `mod 2^32` values.

2.  **Discovering "Circle Paths":** Once the lower 32 bits are correct, we need to find ways to continue hashing more bits without disturbing this partial collision. We achieve this by finding "circle paths" — sequences of bits that, when processed, leave the lower 32 bits of the hash state unchanged. For example, if the current state is `s`, after processing a circle path, the new state `s'` will satisfy `s' ≡ s (mod 2^32)`. These paths are also found using the MITM algorithm, by searching for a path from `target % 2^32` back to itself.

3.  **Progressive Collisions (Stages 2-4):** We then treat these circle paths as a new, more complex "alphabet" (the `valid_paths` in the script). Each "letter" in this alphabet is now a multi-bit sequence that preserves our existing collision.
    *   In Stage 2, we use the 32-bit circle paths as building blocks to find a sequence that collides the next 32 bits, achieving a 64-bit collision in total.
    *   This process is repeated. We find new circle paths that preserve the 64-bit collision and use them to attack the 96-bit state.
    *   Finally, we use 96-bit circle paths to find the final collision for the full 128-bit state.

4.  **Final Message Construction:** After the final stage, we have a complete sequence of bits that hashes to the target value. The script pads this bit sequence to the nearest byte boundary and submits it to get the flag.

## Parameter Selection in `exp.py`

The key parameters in the `exp.py` script are chosen to balance performance and the probability of success:

*   `collision_bit = 32`: This defines the number of new bits to collide in each stage. 32 bits is a reasonable chunk, as finding a collision for this size is computationally feasible with MITM.
*   `search_bits = 36`: This is the initial search depth (number of bits) for the MITM attack. The script adaptively increases this (`try_step_size`) if a path is not immediately found.
*   `MIN_CIRCLE_PATHS = 16`: In each stage, we need to find a sufficient number of circle paths to form a rich enough "alphabet" for the next stage's MITM attack. If fewer than 16 paths are found, the search space for the next stage might be too small, so the script stops.
*   `step_size`: This is the number of "steps" for the MITM attack. It's dynamically calculated based on the number of available `valid_circle_paths`. The formula `ceil(search_bits / log2(len(valid_circle_paths)))` ensures that the search space remains manageable as the circle paths (our alphabet) become more complex.
